## 3. Conditions avant mise en production (bloquantes)

1. **Stripe** : créer les 4 produits/prix (Plus 59 €/an récurrent ; packs 300/600/1 000 € one-shot)
   et renseigner `STRIPE_PLUS_PRICE_ID`, `STRIPE_PACK_{10,50,100}_PRICE_ID` (env local + Vercel).
   Sans eux, les boutons renvoient « Configuration Stripe incomplète » (comportement contrôlé).
2. **Webhook Stripe** : vérifier l'abonnement de l'endpoint aux events `checkout.session.completed`,
   `customer.subscription.updated`, `customer.subscription.deleted`.
3. **Test de bout en bout en mode test Stripe** (cartes 4242…) : abonnement Plus, achat pack,
   annulation → désactivation. *(Non testable localement sans les price IDs.)*
4. **Resend** : domaine `reseauteurs.fr` à vérifier pour que les emails transactionnels
   (plus-active, pack-achete, licence-activee…) partent vers de vrais destinataires.

## 5. Risques suivis (non bloquants)

- **Renouvellement Stripe du Plus** : `plusExpireAt` est rafraîchi par `subscription.updated` ;
  si un event est manqué, le cron `expiration-plus` désactiverait un abonné à tort à J+1 de la
  période. Mitigation existante : Stripe retente les webhooks ; le user est réactivé au prochain
  event. À surveiller en prod (logs cron).
- **Événements de réseauteurs et modération** : ils suivent le même `statut` que les autres
  (`publie` par défaut, suspension admin possible) — cohérent avec la décision « suppression de
  la modération » du 2026-07-08. Si le spam apparaît, réintroduire une file de modération ciblée.
